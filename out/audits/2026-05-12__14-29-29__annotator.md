# Task Audit: Metaspace Exhaustion Hunt in a Hot-Reloading Java Plugin Daemon

**Verdict: Attention**
The main blocker is a realistic tampering path: the evaluated agent can pass by editing `bin/stress`, `bin/integration`, `bin/run-suite`, and `src/main/java/io/harbor/textd/cli/RuntimeDigestRunner.java` instead of fixing the daemon’s reload/shutdown state flow.

---

## What the Task Does
This task ships a Java daemon, `textd`, that hot-swaps plugin revisions from a jar while continuing to serve requests on a fixed two-thread pool. The real bug is a two-part state-flow leak: `ThreadCache.cursorFor(...)` overwrites an old `WorkerHandle` without closing it, so old `TimelineFrame`s stay pinned across refreshes, and `TextDaemon.close()` shuts the pool down without first clearing thread-local handles, so the last revision remains pinned at close. The expected fix is to release stale handles both on revision handoff and on shutdown so the long soak survives under the 64 MB Metaspace cap without changing the shipped text outputs.

## Quality Criteria
### 1. Instructions Clean
Yes. `task/instruction.md` is terse, symptom-first, and aligned with the Task Definition field in the Spec tab. It describes the failure surface and the required outcome without naming the broken classes or repair pattern.

### 2. Hints in the Instructions
No real hint issue. The instruction gives only operational context (“long refresh run”, “constrained CI profile”, “repeated processor swaps”). At the current smart-model pass rate (60% on the newest smart run), I would not flag that as solution narration.

### 3. Environment Realistic
Mostly yes. The environment reads like ordinary Java service code and build scripts, not benchmark scaffolding. The renamed types (`TimelineLedger`, `ThreadCache`, `RuntimeDigestRunner`) are less giveaway-heavy than the earlier lifecycle naming, and the bug is genuinely split across multiple moving parts.

### 4. Environment Integrity
The pre-solve state is present and deterministic, but there is a concrete tampering problem. Hidden tests trust four very small, agent-visible execution surfaces:

- `bin/stress`
- `bin/integration`
- `bin/run-suite`
- `src/main/java/io/harbor/textd/cli/RuntimeDigestRunner.java`

An agent can infer every expected field/value from the shipped source (`OutputFormat`, `HotReloadIntegrationRunner`, current verifier-visible JSON shape) and then stub those surfaces to satisfy all six tests without touching the real leak in `ThreadCache.java` / `TextDaemon.java`. This is a realistic cheat, not a theoretical one: on fast-run Trial 1 (task__KVERPsw), the agent opened `bin/stress` almost immediately, so these runner files are among the first things a solver naturally inspects.

### 5. Agent Has Everything It Needs
Yes. The image contains JDK 17, the full `textd` source tree, build/run scripts, and the shipped plugin sources. The successful smart trials show that the bug is discoverable end-to-end from the prompt plus terminal access.

### 6. Verifier Fairness
Mostly fair. The failure on smart-run Trial 4 (task__2osJegX) and fast-run Trial 1 (task__KVERPsw) is not verifier brittleness: both trials fixed the refresh-path leak but left the close-path leak, and both failed only `test_runtime_digest_reports_stable_run` with `after_close: False`. That is a real missed requirement, not a false negative.

The main fairness caveat is smaller: `test_runtime_digest_reports_stable_run` hard-checks exact worker names (`["textd-worker-1", "textd-worker-2"]`). That is probably fine for this benchmark, but it is slightly more shape-coupled than the rest of the suite.

### 7. Solvability
Solvable from the instruction plus environment. The successful smart trials found both halves of the bug, and the failing smart/fast trials show a coherent near-miss pattern rather than random flailing:

- smart-run Trial 4 (task__2osJegX) identified the `ThreadCache` handoff leak, reproduced the stress failure, and got the digest clean before close, but never patched shutdown, so `after_close` stayed false.
- fast-run Trial 1 (task__KVERPsw) reached the same first-half fix and passed the soak, then stopped after `bin/run-suite` without discovering the close-path cleanup.

That is good evidence that the task is hard-but-fair rather than underspecified.

### 8. Single Focus
Yes. This is one coherent debugging capability: trace a long-run Java reload failure through real code, patch the state flow, and preserve functional behavior.

## Recommendations
- **Must fix** — Add lightweight tamper resistance around the public runner surfaces the verifier currently trusts (`bin/stress`, `bin/integration`, `bin/run-suite`, and `src/main/java/io/harbor/textd/cli/RuntimeDigestRunner.java`), or move more verification below those surfaces; as written, an agent can pass by stubbing outputs instead of fixing the daemon.
- **Worth considering** — In `task/tests/test_outputs.py`, consider checking “same two worker identities before and after” without pinning the exact names `textd-worker-1` / `textd-worker-2`, to reduce shape-coupling if a solver preserves behavior but changes thread naming.
- **Nit** — If `task/environment/textd/build/main-sources.txt` is just a leftover build artifact, consider removing it from the copied environment for cleanliness; it does not affect fairness either way.
