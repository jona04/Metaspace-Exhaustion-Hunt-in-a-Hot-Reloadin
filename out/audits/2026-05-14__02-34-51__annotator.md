# Task Audit: Metaspace Exhaustion Hunt in a Hot-Reloading Java Plugin Daemon

**Verdict: Attention**
The main blocker is verifier fairness and solvability: the task now effectively requires fixing both the daemon leak and a hidden runtime condition for observable class unloading, but the agent-facing prompt and shipped suite only make the daemon leak legible.

---

## What the Task Does
This task ships a Java daemon, `textd`, that hot-swaps plugin revisions while serving requests on a fixed four-thread pool. The code-level bug is still a real multi-part lifecycle leak: `ThreadCache` overwrites a pinned `WorkerHandle`, and `TextDaemon` leaves the last thread-local binding behind at shutdown. The newer task version also requires replayed sessions to show actual class-unload progress after close, which in practice depends on the runtime environment as well as the daemon code. The expected fix path therefore spans reload-state cleanup, thread-local cleanup, and a hidden JVM-layer condition affecting whether `SessionReplayRunner`'s `System.gc()` calls do anything observable.

## Quality Criteria
### 1. Instructions Clean
Mostly yes. `task/instruction.md` is terse, symptom-first, and free of direct repair hints. It frames a long-run runtime failure without naming the broken classes or the intended APIs.

### 2. Hints in the Instructions
No material hint issue. At the current 0% smart-model pass rate, the wording in `task/instruction.md` is not the problem; it is sparse rather than leading.

### 3. Environment Realistic
Mostly. The Java code under `/workspace/textd` reads like normal service code, and the earlier runner-surface tampering problem is now addressed by the integrity hashes in `task/tests/test_outputs.py`. The weaker spot is the hidden Java-launch/runtime layer: requiring observable unloading via a wrapped `java` launcher and `/opt/textd/jvm.cfg` is much less natural than the repo-side bug, especially because that layer is not represented in the agent-facing prompt or the shipped repo scripts.

### 4. Environment Integrity
The pre-solve state is present and the initial code is genuinely broken. The new tamper resistance is also proportionate: hashing the public runners and replay/digest surfaces closes the realistic shortcut the prior audit identified.

The issue is not integrity but hidden state. The environment now contains an extra runtime-level condition outside the repo tree that materially affects pass/fail for unloading, while the visible repo workflow still suggests the task is done once the daemon stays balanced and `bin/run-suite` prints `suite-ok`.

### 5. Agent Has Everything It Needs
Technically yes, but not in a well-signposted way. A very strong debugger could inspect `SessionReplayRunner.java`, notice that `unloaded_after_close` stays at zero even after the daemon is balanced, infer a JVM-level reason, and then go hunting through the runtime launcher setup. In practice, the latest trajectories show agents stopping at the repo-level fix because every visible validation surface says they are done.

### 6. Verifier Fairness
This is the blocker. The hidden verifier checks a stronger invariant than the agent-facing workflow exposes:

- In the newest smart run, Trial 2 (`task__gEwekVM`), Trial 4 (`task__BrgA4aN`), and Trial 5 (`task__4fsw8Un`) all fixed the daemon-side leak, got `before_close=true` and `after_close=true`, passed `bin/run-suite`, and still failed only `test_replay_sessions_unload_classes` with `assert 0 > 0`.
- In the newest fast run on the current checksum, Trial 5 (`task__XLTWCUt`) landed the same code-side fix, saw `suite-ok`, concluded the task was complete, and still failed only `test_replay_sessions_unload_classes` with the same `assert 0 > 0` failure.

That is strong evidence of a hidden requirement, not ordinary near-miss debugging. The core fairness problem is that `bin/run-suite` runs `SessionReplayRunner` but never inspects `/tmp/textd-replay.json`, so the agent-visible “full suite” passes even when the hidden unloading assertion is guaranteed to fail.

### 7. Solvability
Borderline as currently implemented. There is a discoverable path to the daemon leak: both fast and smart models repeatedly found `ThreadCache` and, often, the `TextDaemon` shutdown cleanup. The missing step is the runtime-layer condition needed for unloading to advance. The prompt does not mention class unloading, `bin/run-suite` does not enforce it, and the agent has no explicit reason to inspect the Java launcher after the replay runner already reports clean outputs and clean close state. That pushes the task from “hard but fair” toward “hidden gotcha.”

### 8. Single Focus
Mostly yes. It is still one debugging task around long-lived reload state. The hidden runtime-layer dependency does blur the focus, though, because the agent is no longer just fixing `textd`; it is also implicitly expected to debug the surrounding Java launch environment.
